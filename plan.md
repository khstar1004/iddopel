# 1. 최종 서비스 기획안

## 서비스명

# **ID 도플갱어**

## 슬로건

> **전세계에 나랑 같은 아이디 쓰는 사람 있을까?**
> 

## 제품 한 줄 설명

> 아이디 하나로 전세계 공개 계정 사용 현황을 확인하고, 내 아이디의 희소성·노출도·사칭 가능성을 점수화하는 서비스
> 

## 핵심 포지셔닝

| 구분 | 방향 |
| --- | --- |
| 표면 훅 | “전세계에 나랑 같은 아이디 쓰는 사람 있을까?” |
| 실제 기능 | username 기반 공개 계정 존재 여부 검사 |
| 안전한 명분 | 아이디 희소성, 브랜드명 선점, 계정 노출 점검 |
| 결제 명분 | 전체 리포트, 방치 계정 정리, 사칭 모니터링 |
| 금지 포지션 | 특정인 추적, 신상털이, 동일인 판정 |

# 4. 제품 구조

## 전체 퍼널

```
어그로 유입
→ 아이디 입력
→ 무료 스캔
→ 발견된 공개 후보를 먼저 공개
→ 잠긴 상세 URL과 무료/유료 경계 표시
→ 점수는 마지막 해석 영역에서 공개
→ 발견 개수/국가/카테고리 요약
→ 일부 결과 미리보기
→ 전체 리포트 결제
→ 정기 모니터링 구독
```

## 무료 결과

무료는 **발견된 공개 후보를 먼저 보여주고, 점수·개수·분포는 해석 정보로 마지막에 배치**합니다.

```
전세계 공개 계정 후보: 8개
먼저 공개되는 후보 카드: 3개
잠긴 상세 URL: 5개
한국 서비스 후보: 2개
SNS/커뮤니티 후보: 4개
개발자/포트폴리오 후보: 1개
ID 도플갱어 점수: 72점
확인 실패 사이트: 13%
```

## 유료 결과

```
전체 발견 계정 목록
플랫폼별 URL
위험도 분류
아이디 재사용 위험도
방치 계정 정리 가이드
사칭 신고 가이드
PDF/HTML 리포트
월간 재점검 신청
```

# 5. 점수화 모델

단순히 “계정 8개 발견”보다 **게임처럼 점수화**해야 바이럴이 됩니다.

## 핵심 지표

### 1. ID 희소성 점수

```
적게 발견될수록 높음
```

예:

```
희소성 91점: 전세계에서 비교적 희귀한 아이디예요.
희소성 32점: 여러 플랫폼에서 이미 사용 중인 아이디예요.
```

### 2. 노출도 점수

```
많은 공개 계정이 발견될수록 높음
```

### 3. 사칭 가능성

```
브랜드명/활동명과 동일한 공개 계정이 많으면 상승
```

### 4. 방치 계정 위험

```
오래된 플랫폼, 블로그, 커뮤니티, 프로필성 사이트 발견 시 상승
```

### 5. 국가/카테고리 분포

```
KR 2개
US 3개
Global 4개

SNS 3개
Blog 2개
Developer 1개
Community 2개
```

## 사용자에게 보여줄 결과 문장

```
이 아이디는 전세계에서 8개의 공개 계정 후보가 발견됐어요.
모두 같은 사람이라는 뜻은 아니지만, 아이디가 여러 플랫폼에서 사용 중이라는 신호예요.
```

이 문구가 중요합니다. Maigret은 username 존재 여부를 확인할 수 있지만, 발견된 계정들이 모두 같은 사람인지 확정하는 도구는 아닙니다.

# 7. 토스 인앱 / 모바일 앱 기획

## 앱 포지션

앱은 웹보다 표현을 순화합니다.

웹:

```
내 아이디 털어보기
```

앱/토스:

```
내 아이디 노출 점검
```

토스 앱인토스는 미니앱 출시를 위해 콘솔 등록, 디자인, 개발, 검수, 출시 과정을 안내하고 있고, 토스 UI 스타일을 활용하는 디자인 도구도 제공합니다. (앱인토스 개발자센터) 또한 토스 로그인 사용 시 약관 동의 화면이 노출되고, 파트너사는 서비스 약관과 개인정보 수집·이용 동의 등을 직접 등록해야 합니다. (앱인토스 개발자센터)

## 앱 화면 흐름

# 8. 네이티브 앱 출시 계획

토스 인앱과 별개로 iOS/Android 네이티브 앱을 낼 경우, 개인정보 고지가 특히 중요합니다. Apple은 앱이 수집하는 데이터와 사용 목적을 명확히 밝히는 개인정보처리방침 링크를 요구하고, Google Play도 Data safety 섹션에서 데이터 수집·공유·보호 방식을 공개하도록 요구합니다. (Apple Developer)

## 네이티브 앱은 2차 이후 추천

초기에는 웹과 토스 인앱이 낫습니다.

```
1차: 웹 MVP
2차: 토스 인앱
3차: PWA
4차: 네이티브 앱
```

이유:

```
웹: 빠른 실험, SEO, 공유
토스 인앱: 트래픽과 결제 접근성
PWA: 앱 느낌 구현 가능
네이티브 앱: 심사·개인정보 고지·운영 부담 큼
```

# 11. 기술 아키텍처

## 전체 구조

```
Frontend Web / Toss Mini App
        ↓
Backend API
        ↓
Scan Queue
        ↓
Maigret Worker
        ↓
Result Parser
        ↓
Scoring Engine
        ↓
Database / Report Storage
        ↓
Result UI / PDF Report
```

## 권장 스택

### 웹 프론트

```
Next.js
TypeScript
Tailwind CSS
Framer Motion
React Query
```

### 토스 인앱

```
토스 미니앱 WebView 기반
Next.js 또는 React SPA
Toss SDK
토스 로그인/결제 연동은 2차
```

앱인토스 API는 mTLS 인증서 설정을 요구하고, 통신 구간 암호화와 허용된 서버만 API 호출 가능하도록 설계되어 있습니다. (앱인토스 개발자센터)

### 백엔드

```
Python FastAPI 또는 Node.js NestJS
PostgreSQL
Redis Queue
Celery/RQ 또는 BullMQ
S3 호환 스토리지
Docker
```

### 스캔 워커

```
Python worker
Maigret CLI 또는 library 호출
JSON 결과 저장
Timeout / retry / rate limit
```

Maigret은 Python 프로젝트에 임베딩할 수 있고, CLI뿐 아니라 라이브러리 형태로도 활용 가능하다고 README에 설명되어 있습니다.
